# PRD: GitHub Token 获取个人账户信息与字段管理

## 1. 产品概述

本产品是一个简单的表单页面，用户可以输入自己的 GitHub Personal Access Token，系统通过 GitHub API 获取该用户的个人账户信息，并将信息保存到数据库中。后台使用 Drizzle ORM 管理数据表结构，支持对用户信息字段进行新增和删除。

## 2. 产品目标

- 帮助用户快速验证 GitHub Token 是否可用。
- 自动读取并展示 GitHub 个人账户基础信息。
- 将 GitHub 用户信息结构化保存，便于后续查询和管理。
- 通过 Drizzle 支持字段的新增与删除，方便产品快速迭代数据模型。

## 3. 目标用户

- 需要管理 GitHub 账户信息的个人开发者。
- 正在搭建 GitHub 数据看板或开发者资料管理工具的产品/技术团队。
- 需要用 Drizzle 快速维护数据库字段的全栈开发者。

## 4. 核心使用场景

用户打开表单页面后，输入自己的 GitHub Token，点击提交。系统校验 Token 后，从 GitHub 获取账户信息，例如用户名、头像、邮箱、公司、个人简介、公开仓库数量等，并在页面中展示。如果后续产品需要记录更多字段，开发者可以通过 Drizzle 增加字段；如果某些字段不再需要，也可以通过 Drizzle 删除字段。

## 5. 功能需求

### 5.1 Token 输入表单

页面提供一个简单表单，包含：

- GitHub Token 输入框。
- 提交按钮。
- 加载状态。
- 错误提示。
- 成功后的用户信息展示区。

输入框需要支持隐藏 Token 内容，避免明文展示敏感信息。

### 5.2 Token 校验与账户信息获取

用户提交 Token 后，系统需要：

- 调用 GitHub 用户信息接口获取当前账户资料。
- 判断 Token 是否有效。
- 对无效 Token、权限不足、网络失败等情况给出明确提示。
- 成功后展示账户信息。

建议首期获取字段：

- GitHub 用户 ID
- 登录名 login
- 昵称 name
- 头像 avatar_url
- 邮箱 email
- 公司 company
- 个人网站 blog
- 所在地 location
- 个人简介 bio
- 公开仓库数量 public_repos
- 粉丝数 followers
- 关注数 following
- GitHub 创建时间 created_at

### 5.3 数据库存储

系统需要将获取到的 GitHub 用户信息保存到数据库中。

基础数据表建议命名为 `github_accounts`，首期字段包括：

- `id`
- `github_id`
- `login`
- `name`
- `avatar_url`
- `email`
- `company`
- `blog`
- `location`
- `bio`
- `public_repos`
- `followers`
- `following`
- `github_created_at`
- `created_at`
- `updated_at`

Token 不建议明文保存。若业务必须保存 Token，需要进行加密存储，并明确告知用户。

### 5.4 Drizzle 字段新增

当产品需要记录新的 GitHub 账户字段时，开发者可以通过 Drizzle 完成：

- 在 schema 中新增字段定义。
- 生成 migration。
- 执行 migration 更新数据库结构。
- 在接口返回与页面展示逻辑中增加该字段。

示例需求：新增 `twitter_username` 字段，用于保存用户 GitHub 资料中的社交账号。

验收标准：

- 数据库中成功出现新增字段。
- 新字段可以被写入。
- 页面可以正常展示该字段。
- 原有字段和数据不受影响。

### 5.5 Drizzle 字段删除

当某个字段不再需要时，开发者可以通过 Drizzle 完成：

- 从 schema 中移除字段。
- 生成 migration。
- 执行 migration 更新数据库结构。
- 删除接口和页面中对应字段的引用。

示例需求：删除 `blog` 字段。

验收标准：

- 数据库中该字段被删除。
- 接口不再返回该字段。
- 页面不再展示该字段。
- 系统运行不报错。

## 6. 页面结构

首期只需要一个页面：

页面名称：GitHub 账户信息获取

页面模块：

- 顶部标题：GitHub Account Fetcher
- Token 输入区
- 提交按钮
- 状态提示区
- 用户信息展示区

用户信息展示区建议以列表或信息卡形式展示，包含头像、用户名、简介、仓库数、粉丝数等关键信息。

## 7. 交互流程

1. 用户进入页面。
2. 用户输入 GitHub Token。
3. 用户点击提交。
4. 页面显示加载状态。
5. 系统请求 GitHub API。
6. 如果请求成功，页面展示用户信息，并保存到数据库。
7. 如果请求失败，页面展示错误原因。

## 8. 异常情况

- Token 为空：提示用户请输入 Token。
- Token 无效：提示 Token 不正确或已失效。
- GitHub API 请求失败：提示稍后重试。
- 数据库存储失败：提示账户信息获取成功，但保存失败。
- 字段 migration 失败：提示开发者检查 Drizzle schema 与数据库状态。

## 9. 非功能需求

- Token 输入内容默认隐藏。
- 不在前端日志或服务端日志中打印完整 Token。
- 页面响应时间尽量控制在 3 秒内。
- 接口错误信息需要清晰但不暴露敏感信息。
- 数据库字段变更需要通过 migration 管理，避免手动改库。

## 10. MVP 范围

首个版本只包含：

- 一个 Token 输入页面。
- 获取 GitHub 当前用户基础信息。
- 展示用户信息。
- 保存用户信息到数据库。
- 使用 Drizzle 完成一次字段新增演示。
- 使用 Drizzle 完成一次字段删除演示。

暂不包含：

- GitHub OAuth 登录。
- 多账户团队管理。
- 仓库列表同步。
- 定时同步 GitHub 信息。
- 权限系统。
- Token 管理后台。

## 11. 验收标准

- 用户输入有效 GitHub Token 后，可以看到自己的 GitHub 账户信息。
- 用户输入无效 Token 后，可以看到清晰错误提示。
- 成功获取的信息可以写入数据库。
- Drizzle 可以完成字段新增，并成功读写新字段。
- Drizzle 可以完成字段删除，系统不再引用被删除字段。
- Token 不会被明文暴露在页面、日志或数据库中。

## 12. 后续迭代方向

- 支持 GitHub OAuth 登录，减少用户手动复制 Token 的操作。
- 支持同步仓库、Star、贡献记录等更丰富的数据。
- 支持多次同步并记录历史变化。
- 支持字段配置化展示。
- 增加后台管理页面，查看所有已同步账户。
